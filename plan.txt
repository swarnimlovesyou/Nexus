## The combined product: **Nexus** *(working name)*

> *"Autopilot for LLM usage — observes everything, remembers everything, optimizes over time."*

The core insight from your file is exactly right: Portkey logs data but can't act on it; Contextd remembers but has no real-time decision power. Combined, they form a closed learning loop — the system gets smarter with every single LLM call, without any retraining.

---

## Combined Architecture

### The data flow (as shown in the diagram above)

Every request passes through six stages:

**1. Memory recall (Contextd)** — Before routing to any model, Contextd performs hybrid retrieval (semantic + recency + confidence via RRF fusion). It injects relevant context: user preferences, past prompt outcomes, which models worked well for this query type, prior results.

**2. Smart routing (Portkey)** — Armed with Contextd's memory, the router selects the optimal model not just on static rules but on *learned* performance data. For this query type, did GPT-4o consistently outperform Gemini? Was Claude 3.5 cheaper with equivalent quality? The router knows.

**3. LLM execution** — The enriched prompt goes to whichever model wins the routing decision. The fleet is pluggable: OpenAI, Anthropic, Google, or local/OSS models.

**4. Observation (Portkey)** — Every response is logged: cost, latency, token count, and a quality signal (thumbs up/down, downstream task success, or an LLM-as-judge score in the commercial tier).

**5. Memory storage (Contextd)** — The outcome is written back as a typed memory with confidence, TTL, and conflict detection. This closes the loop: the next routing decision will know what this call produced.

**6. Intelligence layer** — Three emergent capabilities that no competitor can replicate once the loop has run for weeks or months (more on this below).

### `RouterIntelligence.java` — learned routing

```java
// Replaces Portkey's static routing rules with memory-driven decisions.
public class RouterIntelligence {

    private final ContextdClient memory;

    public ModelSelection selectModel(String agentId, String promptHash, 
                                      String queryType) {
        // Pull past outcome memories for this query type
        List<ScoredMemory> outcomes = memory.query()
            .for_(queryType)
            .type(MemoryType.EPISODE)
            .strategy(RetrievalStrategy.HYBRID)
            .minConfidence(0.6)
            .topK(20)
            .retrieve();

        // Score each model on cost × latency × quality
        return outcomes.stream()
            .collect(groupingBy(m -> m.memory().metadata().get("model")))
            .entrySet().stream()
            .map(e -> scoreModel(e.getKey(), e.getValue()))
            .max(Comparator.comparingDouble(ModelScore::composite))
            .map(ModelScore::toSelection)
            .orElse(ModelSelection.DEFAULT);
    }

    private ModelScore scoreModel(String model, List<ScoredMemory> history) {
        double avgQuality = history.stream()
            .mapToDouble(m -> Double.parseDouble(
                m.memory().metadata().get("quality_score")))
            .average().orElse(0.5);
        double avgCost = history.stream()
            .mapToDouble(m -> Double.parseDouble(
                m.memory().metadata().get("cost_usd")))
            .average().orElse(0.01);
        double avgLatency = history.stream()
            .mapToDouble(m -> Double.parseDouble(
                m.memory().metadata().get("latency_ms")))
            .average().orElse(1000);

        // Composite: quality weighted 50%, cost 30%, latency 20%
        double composite = (avgQuality * 0.5)
            - (normalise(avgCost) * 0.3)
            - (normalise(avgLatency) * 0.2);

        return new ModelScore(model, composite, avgCost, avgLatency, avgQuality);
    }
}
```

### `CostIntelligenceEngine.java` — the CFO view

```java
// Answers: "What did you spend? What could you have spent?"
public class CostIntelligenceEngine {

    public CostReport generate(String agentId, Duration window) {
        List<OutcomeMemory> actuals = outcomeStore.findByAgent(agentId, window);

        double actualSpend = actuals.stream()
            .mapToDouble(OutcomeMemory::costUsd).sum();

        // Re-score each call with the cheapest model that met quality threshold
        double optimalSpend = actuals.stream()
            .mapToDouble(o -> cheapestEquivalentCost(o, 0.75))
            .sum();

        return new CostReport(
            actualSpend,
            optimalSpend,
            actualSpend - optimalSpend,   // "savings on the table"
            buildByModelBreakdown(actuals),
            buildByQueryTypeBreakdown(actuals)
        );
    }
}
```

## Database — New Bridge Table

```sql
-- Links every LLM call to a Contextd memory
CREATE TABLE outcome_memories (
    memory_id        UUID PRIMARY KEY REFERENCES memories(memory_id),
    portkey_req_id   VARCHAR(255) NOT NULL,
    agent_id         VARCHAR(255) NOT NULL,
    prompt_hash      VARCHAR(64)  NOT NULL,   -- SHA-256 of normalised prompt
    model_used       VARCHAR(100) NOT NULL,
    cost_usd         DECIMAL(10,8) NOT NULL,
    latency_ms       INTEGER NOT NULL,
    input_tokens     INTEGER,
    output_tokens    INTEGER,
    quality_score    DOUBLE PRECISION,         -- NULL until rated
    query_type       VARCHAR(100),             -- inferred from embedding cluster
    created_at       TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_outcome_agent_model  ON outcome_memories(agent_id, model_used);
CREATE INDEX idx_outcome_agent_type   ON outcome_memories(agent_id, query_type);
CREATE INDEX idx_outcome_prompt_hash  ON outcome_memories(prompt_hash);
```

## Addressing the Hard Problems (from your file)

**1. Evaluation problem ("how do you measure better output?")**
Three-tier approach: (a) explicit thumbs up/down in the CLI/API, (b) downstream task success as a signal (did the agent complete its goal?), (c) LLM-as-judge scoring via `QualityJudge.java` in the commercial tier — sends a stripped, non-identifying version of prompt + response to a cheap model to score on a rubric.

**2. Memory explosion ("what to store vs discard?")**
Contextd's existing TTL + confidence decay solves most of this. Outcome memories default to a 90-day TTL and decay linearly. A `MemoryPruner` job runs nightly and hard-deletes anything below `confidence < 0.15`. The commercial tier adds an LLM-assisted distillation pass: 100 episode memories for the same query type get compressed into a single `SKILL` or `FACT` memory.

**3. Routing logic ("deterministic vs learning-based?")**
Deterministic first. For a cold-start agent with no outcome memories, `StaticRouter` applies Portkey's existing rule-based routing. Once an agent has 20+ outcomes for a query type, `RouterIntelligence` takes over. The handoff is gradual and logged — you can always audit why a routing decision was made.

**4. Latency tradeoff ("more intelligence = slower system?")**
Memory recall and routing selection happen in parallel on the critical path and are capped at 50ms. The Contextd query uses an in-memory embedding cache so the pgvector lookup is a single indexed scan. The `OutcomeCollector` writes to Contextd *after* the response is returned — it is completely off the hot path.

**The compounding moat**
